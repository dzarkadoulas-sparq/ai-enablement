Create a new Alembic migration for: `$ARGUMENTS`.

1. **Generate the revision**

   ```
   uv run alembic revision --autogenerate -m "<$ARGUMENTS>"
   ```

   Alembic picks the filename (timestamp + slug).

2. **Review the autogenerated script** — autogenerate is a starting
   point, not truth. Check:
   - Types match what I actually want (e.g. `TIMESTAMPTZ` not `TIMESTAMP`).
   - `server_default=func.now()` for `created_at` (don't rely on client clock).
   - Indexes on FKs and queried columns.
   - Unique constraints are explicit, not inferred.
   - Autogenerate ignores changes to `CheckConstraint` / comments — add
     those manually.

3. **Non-null on existing tables** — split into steps across migrations:
   (a) add nullable, (b) data backfill in an `op.execute(...)`,
   (c) `alter_column(..., nullable=False)` in a later migration.

4. **Indexes on Postgres**: for production, build concurrently via
   `op.execute("CREATE INDEX CONCURRENTLY ...")` in a migration marked
   `transactional_ddl = False` (see project convention).

5. **Do NOT edit a revision that's been applied above dev.** Create a new
   one. Never `alembic merge` without flagging it to me first.

6. **Check the `downgrade`** — it should either safely revert, or raise
   with a clear message explaining why a downgrade would lose data.

7. **Smoke-test locally**

   ```
   uv run alembic upgrade head
   uv run alembic downgrade -1
   uv run alembic upgrade head
   ```

   Then run the integration tests.

8. **Update SQLAlchemy models** if this migration changes a table the
   app reads/writes. Keep models and migrations in sync in the same PR.

9. **Summary** — revision filename, upgrade/downgrade intent, any
   backfill needed in a later migration, and the next forward migration
   required.
